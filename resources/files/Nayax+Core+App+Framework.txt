Nayax Core App Framework
Introduction
The App Framework enables integration of both internal and external applications into Nayax Core. This provides a unified way to extend the platform's functionality while maintaining security and access control.
Why Use the App Framework?
* Unified Experience: Apps integrate seamlessly into the Nayax Core navigation and UI
* Centralized Access Control: App access is managed through the menu system and role permissions
* Secure Authentication: Apps receive JWT tokens with user identity and scoped permissions
* Context Sharing: Apps can receive context from parent screens (e.g., selected machine)
* Multi-Language Support: Built-in translation system for internationalization
* Role-Based UI Control: Dynamic UI element visibility based on user permissions
How Apps Are Managed
Apps are added through Menu Management (Settings → Menu). Each app menu item can be assigned to specific Roles, controlling which users can access the app. This allows administrators to:
* Add new apps without code changes
* Control access per user role
* Configure app permissions via scopes
* Organize apps within the menu hierarchy

Authentication Token (JWT)

Scopes
Scopes control what resources an app can access. They are injected into the JWT token, limiting or overriding the user's normal permissions.
Scope Types
ValueTypeDescriptionscopeLimitingRestricts app access to specified scopes only (intersection with user permissions)appScopeOverridingGrants app the specified scopes regardless of user's normal permissionsHow Scopes Work
Limiting (scope):
* App can only access resources within both the user's permissions AND the specified scopes
* Use to restrict an app's access even if the user has broader permissions
Overriding (appScope):
* App receives the specified scopes regardless of user's normal permissions
* Use to grant an app specific access that users might not normally have

Receiving Token and Context
Apps receive authentication tokens and context through the following methods:
1. postMessage (External and Internal Apps)
Note: Not available for Fullscreen display type
useEffect(() => {
  const handleMessage = (event: MessageEvent) => {
    const { token, cortexUrl, selection } = event.data;

    if (token) {
      setToken(token);
      setCortexUrl(cortexUrl);

      if (selection?.machineId) {
        setMachineId(selection.machineId);
      }
    }
  };

  window.addEventListener("message", handleMessage);
  return () => window.removeEventListener("message", handleMessage);
}, []);
2. URL Hash Parameter (External and Internal Apps)
/GTraceGenie/index.html#machineId=12345
3. Auth Endpoint (Internal Apps Only)
Internal apps can fetch token from the auth endpoint:
⚠️ Required Header: When calling the Apps/Auth endpoint, you must include the following header:
x-app-type: internal
const resp = await fetch("/core/apps/auth?model=Apps/GTraceGenie", {
  headers: {
    "x-app-type": "internal"
  }
});
const { token, cortexUrl } = await resp.json();

Internal React Apps
App Proxy
Internal apps are served through a proxy endpoint:
/Core/Apps/[serviceName]
This proxy routes to predefined internal service URLs. For example:
* /Core/Apps/ReactApps/GTraceGenie/index.html routes to the react-apps-api service
This abstraction allows internal service URLs to change without affecting menu configurations.
Creating a New Internal React App
Step 1: Copy Template
cd src
cp -r TemplateApp MyNewApp
cd MyNewApp
Step 2: Configure Files
package.json:
{ "name": "my-new-app" }
metadata.json:
{
  "name": "My New App",
  "description": "What your app does",
  "requestFramePermissions": []
}
App.tsx:
const resp = await fetch("/core/apps/auth?model=Apps/MyNewApp", {
  headers: {
    "x-app-type": "internal"
  }
});
Step 3: Add Menu Item
Create menu item in Settings → Menu with appropriate configuration.
Step 4: Assign Roles
Configure which roles can access the app via the Roles tab in menu management.
Step 5: Configure Translations & Permissions (Optional)
For apps requiring multi-language support and role-based UI control:
* See Translations Management for internationalization setup
* See Role-Based UI Visibility Control for dynamic UI element visibility
Step 6: Deploy
Build and deploy via CI/CD pipeline.
