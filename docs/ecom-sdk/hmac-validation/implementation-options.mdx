To integrate HMAC validation, you can either use the official Nayax SDK or implement the logic manually in your preferred language.

## Nayax SDK (C# developers)

C# developers can use the `SDK.Api.Contracts` library, which includes built-in HMAC validation functionality.

1. **Install the library**: Run `Install-Package SDK.Api.Contracts`.
2. **Use the HMAC Service**: Use the `HmacService` class, which handles all HMAC calculation and validation logic automatically.

## Custom implementation (non-C# developers)

If you are not using C#, you should replicate the logic found in the Nayax SDK using these technical specifications:

| Specification | Requirement |
| :--- | :--- |
| **Hmac field name** | `Hmac` This is the JSON field in the notification body. |
| **Fields in signing string** | `NayaxTransactionId`, `MerchantRequestId`, `MachineId`, `RequestTypeName`, `IsApproved`. |
| **Order** | Fields must be in the **exact** order listed above. |
| **Delimiter** | Colon (`:`) |
| **Missing fields** | Use an empty string, but the colon **must** still be included. |
| **RequestType format** | Use the **Enum Name** (e.g., `"Sale"`, `"Auth"`, `"Settlement"`), **not** the integer value. |
| **Boolean format** | `"True"` or `"False"` capital first letter (C# `.ToString()` format). |
| **Key decoding** | 64 hex characters decoded to 32 raw bytes. |
| **Hash output** | Base64. |
| **Exclusions** | Do **not** include `RequestDate`, `Hmac` itself, nested objects, or arrays in the hash. |

## Security best practices
To maintain a high level of security and prevent unauthorized access to your transaction data, adhere to the following guidelines:

| DO | DON'T |
| :--- | :--- |
| Store in environment variables | Hardcode in the source code |
| Use secrets management (Azure Key Vault, etc.) | Commit to version control |
| Decode 64 hex chars to 32 raw bytes | Use the hex string directly as the key |
| Always validate HMAC before processing | Log keys in error messages |
