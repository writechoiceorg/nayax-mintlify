title: "Prepaid Card Operations"
description: "Create prepaid cards, load credit and revalue balances, and search and retrieve card data using the Lynx API."
---

Lynx prepaid cards have two separate stores of value (credit and revalue) and two creation paths with different required fields. Several required fields are undocumented in the spec; this page surfaces them so your requests succeed on the first attempt.

## Create a card

Use `POST /v1/cards` for a simple virtual card, or `POST /v2/cards` for a full-featured prepaid card with credit limits configured at creation.

<Warning>
**`CardTypeID` must be 33**

Always set `CardTypeID: 33` for prepaid cards. Valid values are 31 (Technician), 33 (Prepaid), 34 (Refund), and 30000616 (Discount). Any other value creates the wrong card type or returns an error.
</Warning>

<Warning>
**`CountryID` for cards is 225, not 840**

Card creation endpoints use Nayax internal country codes. `225` is United States. Using the ISO numeric code `840` causes a 500 `country_id_not_valid` error. Actor endpoints use `840`; card endpoints use `225`.
</Warning>

<Tabs>
  <Tab title="v1 — virtual card">
    `POST /v1/cards`

    The `CardPhysicalType` field is required despite not being marked as such in the spec. Omitting it causes a 400 error.

    ```json
    {
      "CardTypeID": 33,
      "CardPhysicalType": 2,
      "CardUniqueIdentifier": "YOUR-UNIQUE-STRING-HERE",
      "Status": 1,
      "CountryID": 225,
      "CardCreditAttributes": {
        "RevalueCashBit": true
      }
    }
    ```

    Include `CardCreditAttributes.RevalueCashBit: true` if the card will ever need revalue operations. There is no way to enable this flag after creation.
  </Tab>
  <Tab title="v2 — prepaid card">
    `POST /v2/cards`

    Two blocks that are undocumented as required will cause the request to fail if omitted: `CardDateRules` and the three credit limit fields inside `CardCreditAttributes`. The error message "CardDetails cant be null" is misleading. The actual cause is a missing `CardDateRules` object (visible in `errorKey: null_CardDateRules`).

    ```json
    {
      "CardTypeID": 33,
      "PhysicalTypeID": 30000528,
      "CardUniqueIdentifier": "YOUR-UNIQUE-STRING-HERE",
      "Status": 1,
      "CountryID": 225,
      "CardDateRules": {
        "ActivationDate": "2026-01-01T00:00:00Z",
        "ExpirationDate": "2028-12-31T00:00:00Z"
      },
      "CardCreditAttributes": {
        "CreditAmountDailyLimit": 100.00,
        "CreditAmountMonthlyLimit": 500.00,
        "CreditAmountMonthlyReload": 500.00,
        "RevalueCashBit": true
      }
    }
    ```

    Note that v2 uses `PhysicalTypeID` (value `30000528`), not the v1 field name `CardPhysicalType`.
  </Tab>
</Tabs>

The creation response includes both the numeric `CardID` and the `DisplayNumber` assigned by Nayax. Store both values, as different endpoints require different identifier types.

## Add and check balances

Credit and revalue are separate stores of value. Credit is the standard spending balance used at machines. Revalue is a separate store used for machine refunds and cashback, and is only available on cards created with `CardCreditAttributes.RevalueCashBit: true`.

All credit and revalue endpoints use `CardUniqueIdentifier` (the string you assigned at creation) in the URL path, not the numeric `CardID`.

### Credit

Add credit to a card:

`POST /v1/cards/{CardUniqueIdentifier}/credit/add`

```json
{
  "CreditAmount": 10.00
}
```

Check the current credit balance:

`GET /v1/cards/{CardUniqueIdentifier}/credit`

### Revalue

Add revalue to a card:

`POST /v1/cards/{CardUniqueIdentifier}/revalue/add`

```json
{
  "RevalueAmount": 5.00
}
```

<Warning>
**Revalue requires `RevalueCashBit: true` at creation**

If the card was created without `CardCreditAttributes.RevalueCashBit: true`, all revalue endpoints return 400 "This Card is not defined as Revalue". There is no way to update this flag after creation. The card must be recreated.
</Warning>

Transfer revalue from one card to another:

`POST /v1/cards/{CardUniqueIdentifier}/revalue/transfer`

Both the source and target card must have `RevalueCashBit: true`.

```json
{
  "TargetCardUniqueIdentifier": "TARGET-CARD-IDENTIFIER",
  "RevalueAmount": 5.00
}
```

## Search and retrieve

There are three ways to look up a card, each using a different identifier.

| Goal | Endpoint | Identifier |
| :-- | :-- | :-- |
| Search with filters | `GET /v1/cards` | query params |
| Look up by your string ID | `GET /v1/cards/{CardUniqueIdentifier}` | string you assigned |
| Look up by Nayax display number | `GET /v1/cards/displayNumber/{DisplayNumber}` | Nayax-assigned number |
| Look up by numeric ID | `GET /v1/cards/{CardID}` | numeric ID from creation |

<Warning>
**`GET /v1/cards` requires at least one filter**

All query parameters are marked optional in the spec, but sending a request with no parameters causes a 400 "Search fields are missing" error. Always include at least one filter, such as `?CardType=33`.
</Warning>

The response from both the search and single-card endpoints is nested, not flat. Access fields through their sub-objects:

```json
{
  "CardDetails": {
    "CardID": 999998796299591,
    "CardUniqueIdentifier": "TEST-CARD-V2-RETEST-001"
  },
  "CardHolderDetails": { },
  "CardCreditAttributes": {
    "CreditBalance": 25.00,
    "RevalueCashBit": true
  }
}
```

Code that accesses top-level fields such as `response.CardID` or `response.CreditBalance` will fail. Use `response.CardDetails.CardID` and `response.CardCreditAttributes.CreditBalance` instead.

### Transaction history

`POST /v1/cards/query` does not accept a standard JSON body. The body must be a bare JSON string containing the base64-encoded SHA1 hash of the card number. Sending an object or empty body causes a 500 error.

To construct the body: compute the SHA1 hash of the card's `DisplayNumber`, base64-encode the hash bytes, and send that string as the entire body.

```json
"dGhpcyBpcyBhbiBleGFtcGxlIGhhc2g="
```

The `Content-Type` header should still be `application/json`. The body is a JSON-encoded string value, not a JSON object.
