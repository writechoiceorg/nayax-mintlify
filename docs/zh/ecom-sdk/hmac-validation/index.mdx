HMAC（基于哈希的消息认证码）是一项可选的安全功能。启用后，Nayax 发送到您服务器的每条商户通知的通知正文中都会包含一个 `Hmac` 字段。

您可以使用它来验证该通知确实来自 Nayax 且未被篡改。

<Note>
  Hmac 字段位于通知正文中，而不在 HTTP 标头中。
</Note>

## 工作原理

要验证收到的通知，请按照以下四个步骤在您的服务器上重新生成签名：

<Steps>
  <Step title="构建签名字符串">
    按照以下确切顺序，从通知中取出这 5 个字段：

    1. `NayaxTransactionId`
    2. `MerchantRequestId`
    3. `MachineId`
    4. `RequestType`（枚举名称，例如 "Sale"、"Auth"、"Settlement"）
    5. `IsApproved`（布尔值，转换为字符串："True" 或 "False"）

    用 `:` 作为分隔符将它们连接起来：
    `{NayaxTransactionId}:{MerchantRequestId}:{MachineId}:{RequestTypeName}:{IsApproved}`

    - 字段缺失：如果通知中缺少某个字段，则该字段使用空字符串代替。分隔符 `:` 仍需保留，切勿跳过某个字段位置。
    - RequestType：不要使用整数值（0、1、2）。请使用枚举中对应的字符串名称。
    - 排除字段：不要包含 `RequestDate`、`Hmac` 本身、嵌套对象或数组（例如 `CardInfo`、`RetryAttempts`）。
  </Step>

  <Step title="准备密钥">
    该密钥是一个 64 字符的十六进制字符串。将其解码为原始字节（每 2 个十六进制字符 = 1 字节），最终得到 32 字节。
  </Step>

  <Step title="运行 HMAC-SHA256">
    - 输入：步骤 1 中的签名字符串，以 UTF-8 字节编码。
    - 密钥：步骤 2 中得到的 32 个原始字节。
    - 输出：32 个原始字节。
  </Step>

  <Step title="编码为 Base64">
    将 32 字节的输出编码为 Base64。这就是出现在 `Hmac` 字段中的值。
  </Step>
</Steps>

## 签名字符串示例

使用以下场景来验证您的字符串拼接逻辑。
所有示例均使用此测试密钥：
`a3f7c2e9d1b8456f0e3a7c9b2d4f6e8a1c3d5e7f9b0a2c4d6e8f0b1c3d5e7f90`

### 示例 1 — 销售通知（存在 `NayaxTransactionId`）

使用下表验证原始通知值是如何转换为签名字符串的：

| 场地                 | 通知中的值                                    | 签名字符串中使用的值                             |
| :----------------- | :--------------------------------------- | :------------------------------------- |
| NayaxTransactionId | `"20000121692"`                          | `20000121692`                          |
| MerchantRequestId  | `"5fbeb1ba-263f-4fe6-a109-642b562020c9"` | `5fbeb1ba-263f-4fe6-a109-642b562020c9` |
| 机器ID               | `"1001316721"`                           | `1001316721`                           |
| RequestType        | `0`（整数）                                  | `Sale`（枚举名称）                           |
| IsApproved         | `true`(布尔值)                              | `True`(大写 T)                           |

签名字符串:

```
20000121692:5fbeb1ba-263f-4fe6-a109-642b562020c9:1001316721:Sale:True
```

预期的 Hmac:

```
uET4OAwxvSN6lwVEwzQ1qRWbMkxo4KR9JbUIcG0qqo0=
```

### 示例 2 — 授权请求（不存在 `NayaxTransactionId`）

此场景演示了如何处理缺失字段。请注意，即使第一个字段为空，也仍需保留分隔符。

| 场地                 | 通知中的值                                    | 签名字符串中使用的值                             |
| :----------------- | :--------------------------------------- | :------------------------------------- |
| NayaxTransactionId | （字段缺失）                                   | （空字符串 — 保留位置）                          |
| MerchantRequestId  | `"e84e9e10-6223-4e45-8da1-243d2d55b25e"` | `e84e9e10-6223-4e45-8da1-243d2d55b25e` |
| 机器ID               | `"1000968111"`                           | `1000968111`                           |
| RequestType        | `1`（整数）                                  | `Auth`（枚举名称）                           |
| IsApproved         | `true`(布尔值)                              | `True`(大写 T)                           |

签名字符串:

```
:e84e9e10-6223-4e45-8da1-243d2d55b25e:1000968111:Auth:True
```

<Note>
  由于 `NayaxTransactionId` 不存在，字符串以冒号开头。切勿跳过某个字段位置。
  该算法要求恰好有 5 个位置，由 4 个冒号分隔。
</Note>

预期的 Hmac:

```
D4Ni+IqJev32uHlNPzz6oW8AFiGyZq7kQ8xh3QyLy8g=
```

## 在您这一端进行验证

1. 接收来自 Nayax 的 POST 通知。
2. 从通知正文中提取这 5 个字段：
   `NayaxTransactionId`、`MerchantRequestId`、`MachineId`、`RequestType`、`IsApproved`。
3. 按照上述步骤 1 中相同的规则构建签名字符串。
4. 将您的密钥从 64 个十六进制字符解码为 32 个原始字节。
5. 以签名字符串（UTF-8 编码）为输入、32 字节为密钥运行 HMAC-SHA256。
6. 对 32 字节的输出进行 Base64 编码。
7. 将您的结果与通知中的 `Hmac` 字段进行比较。

### 验证结果

将您生成的 HMAC 与通知中提供的 HMAC 进行比较后，请按以下操作处理：

| 结果  | 操作                      |
| :-- | :---------------------- |
| 匹配  | 通知是真实的。处理该请求。           |
| 不匹配 | 可能存在篡改或配置错误。拒绝该请求，不要处理。 |

#### HTTP 响应代码

- 返回 200：成功时。
- 返回 401：HMAC 验证失败时。这是有意为之的设计。Nayax 不会针对 401 进行重试。
- 返回 500：仅当您的服务器出现临时性内部错误时。Nayax 最多会重试 3 次。

<Note>
  如果您的 HMAC 实现在开发阶段就是不正确的，重试将无济于事。请修复实现并使用本页的示例进行测试。
</Note>

## 一次性设置

要启用此功能，请完成以下管理和安全任务：

- 生成密钥：集成方生成一个恰好 64 个十六进制字符的密钥。
- 共享密钥：在入驻期间，将密钥安全地与 Nayax 架构师共享。
- 安全存储：将密钥存储为环境变量（切勿硬编码）。
- 环境隔离：生产环境和测试/QA 环境必须使用不同的密钥。

<Warning>
  如果 HMAC 验证在您这端失败，请勿返回 `500` 错误。重试无法修复损坏的 HMAC 实现。请改为返回 `401`。
</Warning>
