# 子流

## 读者始终在线

此配置适用于希望以产品选择（PreSelection 流程）开始交易，但没有方法从外围设备端发起交易的情况——即他们希望交易流程由消费者在 Nayax 设备屏幕上选择产品来发起，然后触发外围设备发送产品详细信息。
在极少数情况下，消费者出示银行卡也可能触发外围设备发送产品选择，但如前所述，这种方式并不常见。

## 始终空闲

交易以消费者在外围设备上选择产品（PreSelection 流程）开始，随后外围设备会发送“Vend request”命令，而无需等待“Begin Session”触发（即“Vend request”）。
此流程在远东地区非常常用。

# 功能

## 传输数据

该命令在交易过程中提供有关交易详情的信息，包括 Nayax 的交易 ID 和消费者的银行卡信息：

信用卡信息：BIN码、PAN哈希值、后四位数字。

对于专有卡 - UID（唯一标识符，卡的完整编号）。

一旦消费者出示银行卡并收到授权结果，卡片详细信息就会自动发送。

<br />

传输数据还提供了一个可选功能，即外围设备可以发送自己的交易 ID，以便将其显示在 SDK 的日志中（该信息没有其他用途）。

<br />

有关传输数据的更多信息，请参阅[“传输数据”](/docs/integrate-pos-device/marshall/additional-resources/marshall-ontransferdata)页面。

## 显示控制

此功能适用于需要 Marshall 主控在发送特定价格的请求之前，在设备上定义不同选择屏幕的外围设备。这样，产品选择就来自 Nayax 设备而非外围设备，这在外围设备没有屏幕或没有方式让消费者自行选择产品的情况下很有帮助。

更多相关信息请参见[“显示控制”](/docs/integrate-pos-device/marshall/additional-resources/marshall-display#/)页面。

## 电子收据

此功能可在交易结算完成后，通过二维码在设备端生成数字收据文件（并通过 SDK 将该文件的链接发送给外围设备）。
要启用该功能，您应在 Nayax Core 中设置以下配置：

• 付款选项卡 -> 交易选项 -> 启用交易结束时的电子收据 - V • 收据按钮显示时长 - 20 • 收据二维码显示时长 - 40 • 销售流程选项 - 动态收据

在旧版本的 SDK 中，URL 以十六进制存储，因此要解析它，您应该[将其转换为 ASCII](https://www.rapidtables.com/convert/number/hex-to-ascii.html) 。

在较新的版本上，它会显示如下内容：

```c C
marshall-c-sdk-demo: received ereceipt: https://ereceipt.nayax.com/qr/2470412964/0123456478901234/2
```

### 如何让电子收据上的产品名称不显示为“未知”？

产品名称取自我们的产品映射表，因此您可以在其中创建产品并命名（以匹配您通过 SDK 发送的产品代码）。请注意，Marshall 不会使用产品映射表中的任何信息，这意味着您可以在其中放置任何信息，这不会影响交易流程，包括价格（唯一与您相关的字段是产品代码名称，它会出现在 Last Sales 和 eReceipt 文件中）

您可以通过转到“产品地图”选项卡，点击“地图”->“添加 BIN”，然后添加产品代码和产品组、PA 代码和 MDB 代码（此代码应与 SDK 中的产品代码匹配），最后点击“保存”，将新产品添加到设备的产品地图中。

## 信息会议

一种信息性的非贩售会话，在消费者出示银行卡后向外围设备提供其卡片详细信息。系统不会向消费者收费（通过“card read successfully”通知），交易以“Vend Denied”结束（因为没有实际授权，也无需进入贩售流程）。此功能主要用于消费者重新识别，即让消费者出示其用于开始交易的同一张卡。通常适用于租赁和电动汽车充电场景。

<Note>
  重要提示：此功能要求外围设备具备屏幕/一种区分贩售会话与信息会话的方式，因为需要由外围设备来判断应发起哪种类型的会话。
</Note>

## 增量授权

增量授权可用于在预估/初始授权金额不足时增加已授权的总金额。
增量授权请求也可能基于对持卡人消费的修订估算。增量授权不会取代原始批准——它们是对先前已授权金额的补充。所有关联的预估授权与增量授权的总和即为该笔交易允许的总金额。

<Danger>
  支持的计费提供商

  截至 2026 年 2 月，支持此功能的计费提供商仅有 Heartland（美国）、Chase（美国）、Credorax（欧洲）、First Data（澳大利亚）、Adyen（美国及欧洲）。
  如果您的设备配置了其他计费提供商，Incremental Authorization 将无法使用。
</Danger>

### 增量授权的工作原理

增量功能有两个“Default Credit”参数——常规参数和“Maximum Default Credit”。
常规默认信用额度是消费者最初获得授权的额度，就像您熟悉的任何 PreAuthorization 交易一样（如果信用额度不足以支付默认信用额度，交易将在此步骤被取消）。
如果消费者取用的商品价值低于默认信用额度金额，则只会对这些商品收费（照常处理）。

#### 增量授权的使用示例：

默认信用额度为 10.00 美元，最大默认信用额度为 50.00 美元：
如果消费者取用的商品价值超过默认信用额度但低于最大默认信用额度（例如，价值 17.00 美元的商品），设备将针对“最终价格”与默认信用额度之间的差额（7.00 美元）进行第二次授权流程：

- 如果第二次授权获得批准，消费者将被收取所需金额（17.00 美元）。在“最近销售记录”中，它将显示如下内容（一行记录，其中结算金额高于授权金额）：

  <Frame>
    <img src="/images/docs/sub-flows-features/image1.png" />
  </Frame>

- 如果第二次授权失败——消费者将只被收取第一次授权的金额，即默认信用额度（10.00 美元）。
  在这种情况下，风险由机器运营商承担，即第二次授权失败造成的损失将由其承担。

如果消费者取用的商品价值超过最大默认信用额度（例如价值 60.00 美元的商品）——机器需要将最终价格发送为最大默认信用额度，否则交易将被取消。
在这种情况下，同样，如果第二次授权未成功，风险由机器运营商承担；如果机器无法向消费者收取超过最大默认信用额度的费用（即至少造成 10.00 美元的损失），风险同样由其承担。
