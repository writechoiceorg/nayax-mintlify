## 支持的专用卡

Nayax 同时支持磁条卡和 RFID 卡。

这支持RFID Nayax读卡器的工作频率为13.56 MHz **，支持ISO 14443 A型和B型卡（例如MiFare Classic和MiFare Plus）以及ISO 15693卡（例如HID iClass）** 。Nayax读卡器支持的最常用、最简单的RFID卡是MiFare Classic 1K（或4K）。

磁条卡也受支持——前提是它们符合非信用卡标准，即卡片数据中**不**包含任何“=”符号，且UID在4-40范围内。如果它们具备上述任何特征，则有可能支持这些特征，但这可能需要额外的集成（[例如Cortina Prepaid](/docs/cortina/prepaid-card/prepaid-card-flows#/) ）。

## Vend被拒——为什么Nayax没有向外围设备提供理由

由于该设备无人值守，我们不希望因银行卡被冻结等情况而让消费者感到尴尬。

## Marshall over ETH：使用同一根线缆同时进行 Marshall 通信以及设备与 Nayax 服务器之间的通信

支持的配置是通过以太网与（Marshall）外围设备和 Nayax 服务器进行通信。路由器的作用是在与外围设备并行的同时提供对外部网络的访问。目前不支持设备通过以太网与外围设备通信、通过 eSIM 与服务器通信的方式，因为这种设置在技术上对 Android 而言不太自然。

## 如何让电子收据上的产品名称不显示为“未知”？

产品名称取自我们的产品地图，因此您可以在其中创建产品并命名（以匹配您通过 SDK 发送的产品代码）。请注意，Marshall 不会使用产品地图中的任何信息，因此您可以随意填写任何内容。这不会影响交易流程，包括价格（唯一与您相关的字段是产品代码名称，它会显示在“上次销售”和电子收据文件中）。

您可以通过转到“产品地图”选项卡，点击“地图”->“添加 BIN”，然后添加产品代码和产品组、PA 代码和 MDB 代码（此代码应与 SDK 中的产品代码匹配），最后点击“保存”，将新产品添加到设备的产品地图中。

## 退款申请

退款是指交易完成结算后，消费者出于某种原因想要退款。Marshall 系统没有退款功能，因为它与客户机器和我们设备之间的通信无关，而是与支付环节有关。如果您无法提供商品，您可以取消交易或不向消费者收费。如果商品发放出现问题，您应该回复“销售失败”而不是“销售成功”。

如果交易已成功完成（消费者已被扣款），而您想要申请退款，可以通过 Nayax Core 进行操作：前往“动态交易监控”页面，选择相关交易并右键单击，选择“申请退款”并确认。Nayax 团队将审核您的申请，审核通过后，您应该会在 1-2 天内收到退款。

## 固件版本更新过程

### VPOST/Onyx：

固件更新由 Nayax Core 发起。固件更新由两部分组成——MAIN 更新和 POS 更新。
对于 MAIN 更新，您会在屏幕顶部看到“gloader”字样，旁边附有百分比。完成后，设备会自行重置并开始下载和更新 POS。在设备重置之前请不要触碰它，因为在 POS 被擦除和更新期间，设备屏幕也会显示为黑屏。完成后，VPOST 会启动电源并进入空闲模式。整个过程通常需要 30-60 分钟（如果使用 ETH 可能需要更长时间），请注意这一点。

### VPOSM 4S：

与 VPOST 和 Onyx 类似，固件更新由 Nayax Core 发起，但与这些设备不同的是，它仅由一个部件组成。

下载过程在后台运行，因此用户可以在下载期间继续使用设备。下载完成后，设备将自动开始安装，并显示相应的安装界面，显示当前的安装进度。

### VPOSM5

与 VPOST/Onyx/VPOSM 4S 不同，固件更新是从一个名为 CasHub 的独立站点发起的。固件更新过程是通过手动更新每个相关应用程序完成的（而不是作为一个整体包）。每个应用程序的固件更新由两个独立部分组成：下载和更新——这意味着一旦下载过程完成，您当地的支持人员需要向设备发送请求，以启动所需应用程序的更新过程。

## 等待收购方批准交易的超时时间

“正在授权，请稍候”的提示会一直显示，直到收到收单机构的响应或超时为止。通常情况下，如果 VPOST 可以与我们的服务器通信（我们的服务器接收收单机构的响应没有问题），发送授权请求并收到响应的整个过程只需几秒钟，通常不超过 5 秒。但是，如果 VPOST 和我们的服务器之间存在任何通信问题（GSM/VPN/MQTT/ETH），请求到达我们的服务器或从我们的服务器收到响应都需要一些时间。我已经就超时问题咨询过技术支持，他们告诉我可以通过“MDB RX Timeout”参数进行配置，默认值为 60 秒。

## “技巧”在于：在一个地点开始交易，在另一个地点完成交易。

为了便于解释，我以购物车租赁为例：

1. 在第一个地点，使用 Nayax 的设备发起交易（从那里取购物车）。
2. 让消费者使用购物车
3. 让消费者将购物车放回他想要的任何轨道（不一定是第一个位置，比如第二个位置）。此时，Nayax 在第二个位置的设备会识别出购物车，并将其 ID 发送到主服务器。主服务器会将购物车 ID 与发起交易的 Nayax 设备进行匹配，并在该服务器上完成交易。但正如您所理解的，在这种情况下，您需要进行服务器管理。

两个要求：

1. 用于管理在哪些位置打开的会话的服务器
2. 区分不同卡片的方法
3. 每条轨道一次最多可容纳 60 辆手推车可供出租（如果超过 60 辆，消费者将无法租用）。

## CPU 要求

所有SDK均无特殊要求。Java**和C# SDK：**

该SDK对CPU和内存的要求极低，只要您的机器能够运行JVM，就能运行该SDK。我们的SDK开发人员估计其内存占用约为50KB RAM，CPU占用几乎为零。（这是基于100行代码，每5毫秒执行一次，并且在接收到串行数据时，每个数据包的执行时间也相差无几。）

**C SDK：**我们的 SDK 开发人员估计其内存占用约为 1KB RAM，每 5 毫秒定时器滴答一次，大约执行 100 行代码（实际机器代码取决于编译器和平台）。他在一块配备 2KB RAM 和 8MHz 控制器的 Atmel AVR 上运行了该 SDK，并且还有充足的内存可供实际应用程序使用。
